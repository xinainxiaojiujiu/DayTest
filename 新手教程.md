  第一章  环境准备与依赖安装
================================================================================

1.1 硬件要求
  - CPU: 4核及以上（推荐8核）
  - 内存: 8GB及以上（推荐16GB）
  - 硬盘: 10GB可用空间
  - 网络: 需能访问阿里云百炼API或Dify平台API

1.2 安装 Python
  步骤1: 访问 https://www.python.org/downloads/
  步骤2: 下载 Python 3.9 或更高版本（推荐 3.11）
  步骤3: 运行安装程序，务必勾选 "Add Python to PATH"
  步骤4: 安装完成后，打开命令提示符验证:
         Win+R → 输入 cmd → 回车 → 输入 python --version
         应显示: Python 3.x.x

1.3 获取项目代码
  将项目文件夹 "Automotive Coating Defect Analysis" 放置到任意目录，
  例如: D:\Trae CN test\Automotive Coating Defect Analysis

1.4 安装依赖包（两种方式任选其一）

  方式一: 一键自动安装（推荐新手使用）
    直接双击运行项目根目录下的 start.bat 文件，
    脚本会自动检测并安装所有缺失的依赖包。

  方式二: 手动安装
    打开命令提示符，进入项目目录:
      cd "D:\Trae CN test\Automotive Coating Defect Analysis"
    
    安装核心依赖:
      pip install fastapi uvicorn[standard] pydantic pydantic-settings
      pip install sqlalchemy[asyncio] aiomysql
      pip install httpx tenacity python-dotenv numpy
    
    可选依赖（按需安装）:
      pip install dashscope          # 使用阿里云百炼时需要
      pip install influxdb-client    # 使用InfluxDB时需要
      pip install chromadb           # 使用向量知识库时需要
      pip install paddleocr          # 使用OCR识别时需要
      pip install pymodbus           # 对接传感器时需要

  第二章  程序部署与配置说明

2.1 配置文件说明
  项目使用 .env 文件管理所有配置。首次运行时，start.bat 会自动从
  .env.example 模板创建 .env 文件。

  配置文件位置: 项目根目录\.env
  配置模板位置: 项目根目录\.env.example

2.2 最小配置（仅需配置以下两项即可启动）

  步骤1: 打开项目根目录下的 .env 文件（用记事本即可）
  步骤2: 选择 LLM 平台并填写对应的 API Key:

  选项A - 使用阿里云百炼（默认）:
    LLM_PLATFORM=dashscope
    DASHSCOPE_API_KEY=sk-xxxxxxxxxxxxxxxxxxxxxxxx
    （API Key 从 https://dashscope.console.aliyun.com/ 获取）

  选项B - 使用 Dify 公司级部署平台:
    LLM_PLATFORM=dify
    DIFY_API_URL=https://aicenter.dongfeng-nissan.com.cn/v1
    DIFY_API_KEY=app-xxxxxxxxxxxxxxxxxxxxxxxx
    （API Key 从 Dify 控制台获取）

  步骤3: 保存 .env 文件

2.3 完整配置项说明

  [应用基础配置]
    APP_NAME    - 应用名称，显示在API文档标题中
    DEBUG       - 调试模式: true=详细日志, false=仅关键日志

  [LLM平台选择]
    LLM_PLATFORM - dashscope(阿里云百炼) 或 dify(Dify平台)

  [阿里云百炼配置]
    DASHSCOPE_API_KEY - API密钥，从阿里云控制台获取
    DASHSCOPE_MODEL   - 模型: qwen-max/qwen-plus/qwen-turbo

  [Dify平台配置]
    DIFY_API_URL       - API地址: https://aicenter.dongfeng-nissan.com.cn/v1
    DIFY_API_KEY       - API密钥，从Dify控制台获取
    DIFY_RESPONSE_MODE - streaming(流式) 或 blocking(阻塞)

  [LLM通用参数]
    LLM_MAX_RETRIES - 调用失败最大重试次数（默认3）
    LLM_RETRY_DELAY - 重试初始延迟秒数（默认2.0）
    LLM_TIMEOUT     - 单次调用超时秒数（默认60）
    LLM_MAX_TOKENS  - 单次最大输出Token数（默认4096）
    LLM_TEMPERATURE - 温度参数0-1，越低越确定（默认0.1）

  [数据库配置 - 均为可选，未配置时使用模拟数据]
    MYSQL_HOST/PORT/USER/PASSWORD/DATABASE - MySQL连接信息
    INFLUX_URL/TOKEN/ORG/BUCKET           - InfluxDB连接信息
    CHROMA_HOST/PORT/COLLECTION           - ChromaDB连接信息
    REDIS_HOST/PORT                       - Redis连接信息

  [外部系统对接 - 均为可选]
    MES_API_BASE_URL/API_KEY - MES系统接口
    ROBOT_HOST/PORT          - 安川机器人控制器
    SENSOR_HOST/PORT         - 环境传感器Modbus网关

3.3 健康检查
  接口: GET /health
  用途: 确认系统是否正常运行
  操作: 在 /docs 页面找到 GET /health → 点击 "Try it out" → "Execute"
  返回: {"status": "healthy", "app": "...", "timestamp": "..."}

3.4 全链路分析（核心功能）
  接口: POST /analyze
  用途: 对不良品执行完整的5阶段分析流程
  操作步骤:
    1. 在 /docs 页面找到 POST /analyze
    2. 点击 "Try it out"
    3. 在 Request body 中输入:
       {
         "defect_id": "DEF-2024-001"
       }
    4. 点击 "Execute"
  
  分析流程（自动执行，无需干预）:
    阶段1 → DataAgent 从MES/机器人/传感器采集数据
    阶段2 → SemanticAgent 解析不良现象为标准化标签
    阶段3 → RootCauseAgent 六维度推理定位根因
    阶段4 → SolutionAgent 生成纠正措施和预防方案
    阶段5 → ClosedLoopAgent 评估有效性并沉淀知识库
  
  返回结果包含:
    - semantic_label: 不良类型、严重程度、置信度
    - root_causes: Top3根因（维度、权重、证据）
    - reasoning_chain: 推理链路摘要
    - solutions: 纠正措施（责任方、优先级、工作量）
    - closed_loop: 验证状态和有效性评分

3.5 单独使用各阶段接口

  仅根因分析（跳过方案生成和闭环验证）:
    POST /analyze/root-cause
    输入: {"defect_id": "DEF-2024-001"}

  仅语义解析（标准化不良描述）:
    POST /analyze/semantic?defect_description=表面出现橘皮
    输入: {"defect_id": "DEF-2024-001"}

  仅生成解决方案:
    POST /solutions/generate
    输入: {"defect_id": "DEF-2024-001"}

  闭环验证（评估已实施方案）:
    POST /closed-loop/verify?defect_id=DEF-2024-001

3.6 数据查询接口

  查询历史不良品记录:
    GET /history/defects?defect_type=缩孔&limit=20

  查看Token消耗统计:
    GET /statistics/token-usage?start_date=2024-01-01&end_date=2024-01-31

  查看知识库案例数量:
    GET /knowledge/count

================================================================================
  第五章  常见问题解决方法
================================================================================

5.1 启动类问题

  Q: 双击 start.bat 后闪退？
  A: 可能原因及解决:
     1. Python未安装 → 安装Python 3.9+并勾选"Add to PATH"
     2. 路径含中文或空格 → 将项目移到纯英文路径下
     3. 权限不足 → 右键 start.bat → "以管理员身份运行"

  Q: 提示 "Python not found"？
  A: Python未添加到系统PATH环境变量。
     解决: 重新运行Python安装程序，选择"Modify"，
           确保勾选"Add Python to environment variables"。

  Q: 启动后报错 "Address already in use"？
  A: 8000端口被占用。
     解决: 关闭占用8000端口的程序，或修改main.py中的端口号。

5.2 配置类问题

  Q: 提示 "请检查 .env 文件中的 API Key"？
  A: API Key未配置或配置错误。
     解决: 打开.env文件，确认DASHSCOPE_API_KEY或DIFY_API_KEY已填写正确。
           注意: API Key不要加引号。

  Q: 如何切换LLM平台？
  A: 编辑.env文件，修改 LLM_PLATFORM 的值:
     LLM_PLATFORM=dashscope  → 使用阿里云百炼
     LLM_PLATFORM=dify       → 使用Dify平台
     修改后重启系统生效。

  Q: 没有MySQL/InfluxDB/ChromaDB怎么办？
  A: 这些数据库都是可选的。系统检测到数据库未连接时，
     会自动使用内置的模拟数据运行，不影响核心分析功能。
     模拟数据包含典型的涂装工艺参数，可用于功能测试。

5.3 运行类问题

  Q: 调用 /analyze 接口返回500错误？
  A: 排查步骤:
     1. 检查终端日志中的具体错误信息
     2. 确认API Key是否正确（常见错误: 复制时多了空格）
     3. 确认网络能访问LLM平台API
     4. 查看返回的detail字段中的提示信息

  Q: 分析结果中根因不准确？
  A: 可能原因:
     1. LLM温度参数过高 → 在.env中降低 LLM_TEMPERATURE（如0.05）
     2. 知识库案例不足 → 积累更多有效案例后准确率会提升
     3. 不良描述不够详细 → 提供更具体的缺陷描述

  Q: OCR识别不工作？
  A: PaddleOCR依赖较大，首次使用需下载模型文件。
     如果不需要OCR功能，可以不安装paddleocr和paddlepaddle。
     系统在OCR不可用时会跳过图片识别步骤。

5.4 性能类问题

  Q: 分析速度慢？
  A: 优化建议:
     1. 使用更快的模型: DASHSCOPE_MODEL=qwen-turbo
     2. 减少LLM_MAX_TOKENS（如2048）
     3. 关闭DEBUG模式: DEBUG=false
     4. 确保网络延迟低

  Q: Token消耗过高？
  A: 控制方法:
     1. 降低LLM_MAX_TOKENS
     2. 使用更经济的模型（qwen-turbo）
     3. 定期查看 GET /statistics/token-usage 监控消耗

  第六章  功能扩展与自定义配置指导
================================================================================

6.1 添加新的不良类型
  编辑文件: src/agents/semantic_agent.py
  找到 DEFECT_TAXONOMY 字典，在对应分类下添加新类型:
    DEFECT_TAXONOMY = {
        "外观缺陷": ["缩孔", "橘皮", "流挂", "颗粒", "气泡", "针孔", "你的新类型"],
        ...
    }

6.2 调整SOP标准范围
  编辑文件: src/agents/root_cause_agent.py
  找到 SOP_RANGES 字典，修改各参数的min/max值:
    SOP_RANGES = {
        "voltage": {"name": "喷涂电压", "min": 60, "max": 80, "unit": "kV"},
        ...
    }
  修改后系统会自动按新标准检测工艺参数违规。

6.3 对接真实MES系统
  编辑文件: src/connectors/mes_connector.py
  修改 get_product_info 方法中的API调用逻辑，
  将模拟数据替换为真实的HTTP请求。

  配置 .env 文件:
    MES_API_BASE_URL=http://你的MES系统地址/api/v1
    MES_API_KEY=你的MES_API密钥

6.4 对接真实安川机器人
  编辑文件: src/connectors/robot_connector.py
  修改 _read_robot_param 方法中的通讯协议，
  适配实际机器人的寄存器地址和通讯格式。

  配置 .env 文件:
    ROBOT_HOST=192.168.1.100    ← 机器人控制器IP
    ROBOT_PORT=10000            ← 通讯端口

6.5 对接真实环境传感器
  编辑文件: src/connectors/sensor_connector.py
  修改寄存器映射表 REGISTER_MAP 以匹配实际传感器配置。

  配置 .env 文件:
    SENSOR_HOST=192.168.1.200   ← Modbus网关IP
    SENSOR_PORT=502             ← Modbus TCP端口

6.6 自定义LLM提示词
  每个Agent的LLM调用提示词都在对应文件中定义。
  以根因分析Agent为例，编辑:
    src/agents/root_cause_agent.py
    
  找到 _build_analysis_prompt 方法中的 system_prompt 变量，
  根据需要调整提示词内容以适配不同的分析需求。

6.7 添加新的API接口
  编辑文件: src/main.py
  参照现有接口的格式添加新的路由:
    @app.get("/your-new-endpoint")
    async def your_new_function():
        # 你的逻辑
        return {"result": "..."}

6.8 扩展为Web界面
  系统已启用CORS跨域支持，可直接被前端页面调用。
  在 src/main.py 中可添加静态文件服务:
    from fastapi.staticfiles import StaticFiles
    app.mount("/static", StaticFiles(directory="static"), name="static")
  
  然后创建 static/ 目录放置HTML/CSS/JS文件即可。
